# PAN + Home Assistant Integration

**Read this before building any smart home device control into PAN directly.**
Home Assistant already solved this problem for 3,000+ devices. PAN absorbs it.

---

## Core Philosophy

Do not rebuild what Home Assistant has already spent years perfecting. Home Assistant is Apache 2.0 licensed, owned by the Open Home Foundation (a non-profit that cannot be sold or acquired), and is the #1 open source project on GitHub by contributors with 21,000+ active contributors. PAN wraps it, consumes its event stream, and lets it handle all device complexity.

PAN's job: be the brain, the memory, the AI, and the interface.
Home Assistant's job: talk to every device on the network.

## The Three PAN Installers

PAN has three distinct installation targets. Home Assistant runs on the Hub only.

### 1. PAN Hub (Server)

Installed on one machine — your primary always-on PC (TedProDesk1).

**Runs:**
- PAN service (Carrier + Craft)
- Home Assistant (Docker container)
- SQLCipher database
- MCP server
- Dream cycle
- All background services (Steward, Orphan Reaper, Tool Scout)

**Controls:** Everything. All clients report to the hub. Hub has full authority over all connected devices.

**Install:** Full PAN install script. Sets up WinSW service, Docker, Home Assistant, all dependencies.

### 2. PAN Client (PC)

Installed on any additional computer — TedProDesk2, TedProDesk3, laptops, any Windows/Linux machine.

**Runs:**
- PAN client process (WebSocket connection to hub)
- Tauri wrapper (opens apps in PAN chrome)
- Local command executor
- TPM 2.0 signing for local events

**Does NOT run:** Its own database, its own Home Assistant, its own Carrier.

**Relationship to hub:** Like SSH — hub sends shell commands, the client executes them. Hub has full control. Client is an executor and observer. Everything captured on the client (keystrokes, app usage, screen content, BLE devices nearby) gets signed with the client's TPM key and sent to hub.

**Install:** Single command from hub generates an invite token. Client machine runs one-liner, registers itself, appears in hub device list immediately.

```bash
# Generated by hub, run on client machine (Windows)
irm https://100.x.x.x:7777/install/TOKEN | iex

# Linux
curl -s https://100.x.x.x:7777/install/TOKEN | bash
```

### 3. PAN Mobile (Android)

Installed on Android phones.

**Runs:**
- PAN Android app (foreground service, always-on)
- Voice assistant (always-on STT via Google Streaming Speech Recognition)
- BLE receiver (pendant data over Bluetooth)
- Sensor collection (GPS, accelerometer, mic, camera)
- Titan M2 signing for all events
- PAN terminal (built — rough but functional, gives hub shell access to phone)
- Tailscale VPN via embedded tsnet

**Relationship to hub:** Same authority model as PC client. Hub can send shell commands to the phone terminal, execute scripts, control device features programmatically. All phone terminal sessions captured as events in PAN database.

## Home Assistant on the Hub

### Installation (Docker)

```yaml
# Add to docker-compose.yml on TedProDesk1
services:
  homeassistant:
    container_name: homeassistant
    image: ghcr.io/home-assistant/home-assistant:stable
    volumes:
      - ./ha-config:/config
      - /etc/localtime:/etc/localtime:ro
    restart: unless-stopped
    privileged: true
    network_mode: host
```

Home Assistant runs on port 8123. PAN Dashboard opens it in a Tauri window with PAN chrome — same pattern as Discord, Chrome, any other wrapped app. Users never leave PAN.

### PAN Settings Required

```javascript
// Add to PAN settings table
{
  ha_url: 'http://localhost:8123',
  ha_token: 'long-lived-access-token-from-ha-settings',
  ha_enabled: true
}
```

Generate the long-lived token in Home Assistant under Profile → Long-Lived Access Tokens.

## Event Ingestion — HA → PAN Database

Home Assistant exposes a WebSocket API that streams every state change in real time. PAN subscribes to this stream and writes every event into the PAN events table with TPM signing.

```javascript
// service/src/routes/homeassistant.js
const WebSocket = require('ws')

function connectToHA(haUrl, haToken) {
  const ws = new WebSocket(`ws://${haUrl}/api/websocket`)

  ws.on('open', () => {
    ws.send(JSON.stringify({ type: 'auth', access_token: haToken }))
  })

  ws.on('message', async (raw) => {
    const msg = JSON.parse(raw)

    if (msg.type === 'auth_ok') {
      // Subscribe to all state changes
      ws.send(JSON.stringify({
        id: 1,
        type: 'subscribe_events',
        event_type: 'state_changed'
      }))
    }

    if (msg.type === 'event' && msg.event?.event_type === 'state_changed') {
      const { entity_id, new_state, old_state } = msg.event.data

      await insertSignedEvent({
        type: 'home_assistant',
        data: {
          entity_id,                           // e.g. 'light.living_room'
          domain: entity_id.split('.')[0],     // 'light', 'switch', 'sensor'
          new_state: new_state?.state,
          old_state: old_state?.state,
          attributes: new_state?.attributes,
        },
        device_id: HUB_DEVICE_ID,
        trust_origin: 'home_assistant'
      })
    }
  })

  // Auto-reconnect on disconnect
  ws.on('close', () => setTimeout(() => connectToHA(haUrl, haToken), 5000))
}
```

### What Gets Captured

Every state change across all connected devices:

| Domain | Examples | Why it matters for PAN |
|--------|---------|------------------------|
| `light` | On/off, brightness, color | Presence inference, routine detection |
| `switch` | Plug on/off, appliance state | Power usage, activity patterns |
| `sensor` | Temperature, humidity, motion | Environmental context |
| `binary_sensor` | Motion detected, door open/closed | Presence, security events |
| `person` | Home/away status | Who is where |
| `device_tracker` | Phone GPS (home/away) | Presence detection |
| `climate` | Thermostat state, temperature | Comfort, routine |
| `media_player` | TV on/off, what's playing | Activity context |
| `lock` | Door locked/unlocked | Security, presence |
| `camera` | Motion detected | Security events |
| `energy` | Power consumption per device | Usage patterns, dividend data |

## Bluetooth Device Layer

Home Assistant handles BLE device connectivity far better than custom code:

- **Reconnection logic already solved** — devices that lose WiFi/BLE reconnect automatically without user intervention. No more powering devices on three times to reset them.
- **Local control** — no cloud in the path. Devices connect directly to the HA instance on the local network. If the device drops, HA retries locally. Cloud going down does not affect your devices.
- **Bluetooth proxy support** — any ESP32 running HA Bluetooth proxy extends BLE range throughout the house. The pendant itself could act as a BLE proxy when docked.
- **Passive BLE scanning** — detects any BLE device nearby without pairing. Enables room-level presence detection from signal strength.

**Why devices lose connection constantly (and why HA fixes it):**
The reason smart devices constantly drop connection is they use cloud relay — device talks to manufacturer's cloud, cloud talks to your app. When the cloud has issues, or the device's Wi-Fi drops briefly, the whole chain breaks and manual reconnection is required. Home Assistant uses local control — no cloud in the path. Device talks directly to HA on your LAN. Dramatically more reliable, works without internet.

**For PAN's pendant specifically:**
The pendant broadcasts BLE. Any PAN client PC or hub running Home Assistant Bluetooth integration can detect the pendant's presence and signal strength — giving PAN room-level location awareness of the pendant (and therefore the user) without GPS.

- "Is the front door locked?"
- "What's the temperature in the bedroom?"
- "Is the oven on?"

- "Lock the front door" → PAN → HA → door lock
- "Set the bedroom to sleep mode" → PAN triggers HA scene
- "Turn on the coffee machine" → PAN → HA → Tapo plug

### Cross-system automations (PAN + HA together):
- Pendant detects you walking toward a room (BLE signal strength) → HA turns on lights before you arrive
- Phone GPS says you're 5 minutes away → HA turns on heating, unlocks door
- Pendant BME688 detects CO spike → HA opens smart windows + PAN sends urgent alert to all devices
- Pendant thermometer detects fever pattern → HA adjusts thermostat, PAN logs health event
- Dream cycle analyzes weekly energy data → suggests which devices to schedule off-peak
- HA motion sensor detects no motion for 2 hours → PAN marks user as away, adjusts context

## What Home Assistant Does NOT Replace in PAN

| PAN feature | HA equivalent | Verdict |
|-------------|--------------|---------|
| Pendant sensors (thermal, spectrometer, EMF, UV) | None | PAN only |
| Always-on voice assistant with AI memory | HA Assist (limited, no persistent memory) | PAN is better |
| AI memory, dream cycle, context injection | None | PAN only |
| Data dividends and cryptographic signing | None | PAN only |
| Cross-device event timeline and FTS5 search | None | PAN only |
| Carrier/Craft zero-downtime runtime | None | PAN only |
| MCP server for Claude Code integration | None | PAN only |

HA handles the device layer. PAN handles everything above it.
